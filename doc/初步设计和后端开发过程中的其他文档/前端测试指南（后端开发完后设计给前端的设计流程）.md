# 前端测试指南

本文档供前端完成后设计和执行联调测试使用。接口字段以 `前端API接口文档.md` 为准，本文重点说明：后端如何本地启动、如何准备测试数据、前端页面和交互应验证哪些场景。

如果需要交付给前端人员的一站式开发与测试入口文档，请优先阅读 `doc/前端开发与测试交付文档.md`。

## 1. 测试目标

前端联调测试建议覆盖以下目标：

- 页面能正确读取后端公开数据：封面、信息页、文章列表、文章详情、分类、标签、评论。
- 登录态和权限行为正确：游客、管理员、未登录三种状态展示和跳转符合预期。
- 管理后台能完成核心内容管理：文章、分类、标签、评论、游客、配置、图片上传。
- 错误状态可被前端友好处理：401、403、404、409、400、429。
- 前端不会展示或依赖任何密码明文、`passwordHash` 或数据库内部字段。

## 2. 后端本地启动方式

### 2.1 推荐方式：Docker Compose

前置条件：

- Docker 可用。
- Docker Compose v2 可用，即命令 `docker compose version` 能正常执行。

启动：

```bash
cd /Users/samlee/Desktop/Code/Blog-backend
docker compose up --build
```

默认服务：

| 服务 | 地址 |
| --- | --- |
| 后端 API | `http://localhost:8080` |
| MySQL | `localhost:3306` |
| Redis | `localhost:6379` |
| 上传文件 URL | `http://localhost:8080/uploads/...` |

默认管理员账号：

```text
username: admin
password: password123
```

停止：

```bash
docker compose down
```

清空数据库和 Redis 数据卷后重来：

```bash
docker compose down -v
docker compose up --build
```

注意：`down -v` 会删除本地测试数据。

### 2.2 替代方式：无 Docker Compose 时手动启动

如果本机没有 `docker compose`，但有 Docker，可手动启动依赖容器。

1. 启动 MySQL 和 Redis：

```bash
docker rm -f blog-backend-test-mysql blog-backend-test-redis 2>/dev/null || true

docker run -d --name blog-backend-test-mysql \
  -e MYSQL_DATABASE=blog \
  -e MYSQL_USER=blog \
  -e MYSQL_PASSWORD=blog_password \
  -e MYSQL_ROOT_PASSWORD=root_password \
  -p 3307:3306 \
  mysql:8.4

docker run -d --name blog-backend-test-redis \
  -p 6380:6379 \
  redis:7.4
```

2. 等待 MySQL ready：

```bash
docker exec blog-backend-test-mysql mysqladmin ping -uroot -proot_password --silent
```

看到 `mysqld is alive` 后继续。

3. 打包并启动后端：

```bash
mvn package -DskipTests

SERVER_PORT=18080 \
SPRING_DATASOURCE_URL='jdbc:mysql://localhost:3307/blog?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai&useSSL=false&allowPublicKeyRetrieval=true' \
SPRING_DATASOURCE_USERNAME=blog \
SPRING_DATASOURCE_PASSWORD=blog_password \
SPRING_DATA_REDIS_HOST=localhost \
SPRING_DATA_REDIS_PORT=6380 \
APP_UPLOAD_ROOT=./data/test-uploads \
APP_ADMIN_USERNAME=admin \
APP_ADMIN_PASSWORD=password123 \
java -jar target/blog-backend-0.0.1-SNAPSHOT.jar
```

此方式下后端地址为：

```text
http://localhost:18080
```

4. 清理：

```bash
docker rm -f blog-backend-test-mysql blog-backend-test-redis
```

如果后端仍在运行，停止对应终端进程即可。

## 4. 前端联调基础配置

建议前端项目使用环境变量配置 API 根地址：

```text
VITE_API_BASE_URL=http://localhost:8080
```

或：

```text
NEXT_PUBLIC_API_BASE_URL=http://localhost:8080
```

如果使用手动 Docker 方式启动后端，则改为：

```text
http://localhost:18080
```

请求封装建议：

- 所有响应先检查 `code`。
- `code === "SUCCESS"` 时读取 `data`。
- `401` 或 `code === "UNAUTHORIZED"`：清理 token，跳转登录页。
- `403` 或 `code === "FORBIDDEN"`：展示无权限提示。
- `VALIDATION_ERROR`、`CONFLICT`、`UPLOAD_ERROR`：展示后端 `message`。
- 分页组件使用 `data.page`、`data.size`、`data.total`、`data.pages`。

## 5. 测试数据准备

以下命令假设后端地址为 `http://localhost:8080`。如果你使用 `18080`，请替换 `BASE_URL`。

```bash
BASE_URL=http://localhost:8080
```

### 5.1 管理员登录并保存 token

```bash
ADMIN_TOKEN=$(curl -s -X POST "$BASE_URL/api/auth/login" \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"password123"}' | jq -r '.data.token')

echo "$ADMIN_TOKEN"
```

期望 token 非空。

### 5.2 创建分类

```bash
CATEGORY_ID=$(curl -s -X POST "$BASE_URL/api/admin/categories" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"name":"Tech","slug":"tech","description":"Technical notes","sortOrder":1}' | jq -r '.data.id')

echo "$CATEGORY_ID"
```

### 5.3 创建标签

```bash
TAG_ID=$(curl -s -X POST "$BASE_URL/api/admin/tags" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"name":"Spring","slug":"spring"}' | jq -r '.data.id')

echo "$TAG_ID"
```

### 5.4 创建已发布文章

```bash
curl -s -X POST "$BASE_URL/api/admin/articles" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H 'Content-Type: application/json' \
  -d "{
    \"title\":\"First Article\",
    \"slug\":\"first-article\",
    \"summary\":\"Article summary\",
    \"contentMarkdown\":\"# First Article\",
    \"categoryId\":$CATEGORY_ID,
    \"tagIds\":[$TAG_ID],
    \"status\":\"PUBLISHED\"
  }" | jq
```

### 5.5 注册游客并评论

```bash
curl -s -X POST "$BASE_URL/api/auth/guest/register" \
  -H 'Content-Type: application/json' \
  -d '{"username":"visitor01","password":"password123","nickname":"Visitor"}' | jq

GUEST_TOKEN=$(curl -s -X POST "$BASE_URL/api/auth/login" \
  -H 'Content-Type: application/json' \
  -d '{"username":"visitor01","password":"password123"}' | jq -r '.data.token')

curl -s -X POST "$BASE_URL/api/articles/first-article/comments" \
  -H "Authorization: Bearer $GUEST_TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"content":"<b>Nice article!</b>"}' | jq
```

期望评论内容返回：

```text
&lt;b&gt;Nice article!&lt;/b&gt;
```

### 5.6 上传图片

仓库中已有测试图片：

```bash
curl -s -X POST "$BASE_URL/api/admin/media/images" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -F usageType=COVER \
  -F file=@doc/image/cover.png | jq
```

返回的 `data.url` 可用于封面、信息页或文章 Markdown。

## 6. 前端页面测试设计

### 6.1 封面页

接口：

- `GET /api/cover`

验证点：

- 首屏标题、副标题、头像、背景图、链接渲染正确。
- `backgroundImageUrl`、`avatarImageUrl` 为 `null` 时页面有合理占位或降级。
- `links` 为空数组时页面不报错。
- 链接按 `sortOrder` 展示；如果前端负责排序，需要覆盖乱序数据。

### 6.2 信息页

接口：

- `GET /api/profile`

验证点：

- `displayName`、`bio`、`email`、`location` 正常展示。
- `contentMarkdown` 能正确渲染。
- `socialLinks` 为空数组时页面不报错。
- 可选字段为 `null` 时不展示空白坏样式。

### 6.3 文章列表页

接口：

- `GET /api/articles`
- `GET /api/categories`
- `GET /api/tags`

验证点：

- 默认分页展示已发布文章。
- 分类筛选 `categorySlug` 生效。
- 标签筛选 `tagSlug` 生效。
- 关键词 `keyword` 生效。
- 空列表展示正确。
- 翻页时 `page`、`size`、`total`、`pages` 使用正确。
- 文章封面为空时有降级展示。
- `status` 虽然会返回，但前台不应展示草稿/隐藏文章。

建议用例：

| 场景 | 输入 | 期望 |
| --- | --- | --- |
| 默认列表 | 无筛选 | 返回已发布文章 |
| 分类筛选 | `categorySlug=tech` | 只展示该分类文章 |
| 标签筛选 | `tagSlug=spring` | 只展示该标签文章 |
| 关键词搜索 | `keyword=First` | 标题或摘要匹配 |
| 无结果 | 不存在的筛选值 | 空状态 |

### 6.4 文章详情页

接口：

- `GET /api/articles/{slug}`
- `GET /api/articles/{slug}/comments`
- `POST /api/articles/{slug}/comments`

验证点：

- Markdown 正文正常渲染。
- `contentHtml` 为 `null` 时前端使用 `contentMarkdown`。
- 分类、标签、发布时间、浏览量展示正确。
- 不存在的 slug 展示 404 页面或空状态。
- 评论列表分页展示。
- 未登录发表评论时跳转登录或提示登录。
- 游客登录后可评论。
- 评论中的 HTML 内容按文本展示，不执行 HTML。

### 6.5 登录页

接口：

- `POST /api/auth/login`
- `POST /api/auth/guest/register`
- `GET /api/auth/me`
- `POST /api/auth/logout`

验证点：

- 管理员登录后进入后台。
- 游客登录后只能评论，不能进入后台。
- 错误密码显示错误提示。
- 禁用游客登录失败。
- 连续输错触发限流时展示合适提示。
- 刷新页面后能通过本地 token 恢复用户状态。
- 退出登录后 token 清除，受保护页面不可访问。

限流测试：

- 同一用户名和 IP，15 分钟内失败 5 次后返回 `RATE_LIMITED`。
- 前端不应无限重试登录接口。

### 6.6 后台文章管理

接口：

- `GET /api/admin/articles`
- `POST /api/admin/articles`
- `GET /api/admin/articles/{id}`
- `PUT /api/admin/articles/{id}`
- `DELETE /api/admin/articles/{id}`
- `POST /api/admin/media/images`

验证点：

- 未登录访问后台接口返回 401。
- 游客访问后台接口返回 403。
- 管理员可查看草稿、发布、隐藏文章。
- 创建草稿成功。
- 直接创建已发布文章后前台可见。
- 草稿更新为已发布后前台可见。
- 编辑已发布文章不丢失原发布时间。
- 删除文章后前台详情返回 404。
- slug 重复时显示冲突提示。
- 分类或标签不存在时显示错误提示。
- Markdown 编辑器插入上传图片 URL 后保存成功。

建议文章状态用例：

| 状态 | 前台列表 | 前台详情 | 后台列表 |
| --- | --- | --- | --- |
| `DRAFT` | 不显示 | 404 | 显示 |
| `PUBLISHED` | 显示 | 200 | 显示 |
| `HIDDEN` | 不显示 | 404 | 显示 |
| 已删除 | 不显示 | 404 | 不显示 |

### 6.7 后台分类和标签管理

接口：

- `GET /api/categories`
- `GET /api/tags`
- `POST /api/admin/categories`
- `PUT /api/admin/categories/{id}`
- `DELETE /api/admin/categories/{id}`
- `POST /api/admin/tags`
- `PUT /api/admin/tags/{id}`
- `DELETE /api/admin/tags/{id}`

验证点：

- 当前没有专用后台分类/标签列表接口，后台列表可复用公开接口。
- 创建分类/标签成功后，文章编辑页选项刷新。
- 分类 slug 重复返回 `CONFLICT`。
- 标签 name 或 slug 重复返回 `CONFLICT`。
- 分类下存在文章时删除分类返回 `CONFLICT`。
- 删除标签后文章详情不再展示该标签。

### 6.8 后台评论管理

接口：

- `GET /api/admin/comments`
- `PATCH /api/admin/comments/{id}/status`
- `DELETE /api/admin/comments/{id}`

验证点：

- 支持按状态、文章 ID、用户名筛选。
- 隐藏评论后前台评论列表不再展示。
- 删除评论后前台和后台列表均不展示。
- 后台能看到 IP 和 User-Agent。

### 6.9 后台游客管理

接口：

- `GET /api/admin/guests`
- `PATCH /api/admin/guests/{id}/password`
- `PATCH /api/admin/guests/{id}/status`

验证点：

- 游客列表不展示密码相关字段。
- 重置密码后旧密码不能登录，新密码可登录。
- 重置密码后旧 token 失效。
- 禁用游客后无法登录，已有 token 失效。
- 启用游客后可重新登录。

### 6.10 后台站点配置

接口：

- `GET /api/admin/cover`
- `PUT /api/admin/cover`
- `GET /api/admin/profile`
- `PUT /api/admin/profile`
- `GET /api/admin/system-configs`
- `PUT /api/admin/system-configs/{configKey}`

验证点：

- 更新封面标题、背景图、头像、链接后，前台封面页刷新可见。
- 更新信息页内容后，前台信息页刷新可见。
- `links` / `socialLinks` 为空数组时保存和展示正常。
- 更新 `comment.defaultStatus` 为 `PENDING` 后，新评论不应出现在前台评论列表。
- 更新 `upload.maxFileSizeMb` 后，大文件上传限制变化。

### 6.11 图片上传

接口：

- `POST /api/admin/media/images`
- `GET /api/admin/media/images`
- `DELETE /api/admin/media/images/{id}`

验证点：

- 管理员可上传 `jpg/jpeg/png/webp/gif`。
- 游客或未登录用户上传返回 401/403。
- 非图片文件返回 `UPLOAD_ERROR`。
- 文件扩展名和 MIME 不匹配返回 `UPLOAD_ERROR`。
- 超出大小限制返回 `UPLOAD_ERROR`。
- 上传成功后返回 `url`，浏览器可访问。
- 删除图片记录后，图片列表不再展示该记录。
- 已在文章中引用的图片 URL 不会自动从 Markdown 移除。

## 8. 错误处理测试

### 8.1 参数校验

用例：

- 注册用户名少于 3 位。
- 注册用户名含空格或中文。
- 密码少于 6 位。
- 文章标题为空。
- 评论内容为空或超过 2000 字符。
- 分页 `page=0` 或 `size=101`。

期望：

```json
{
  "code": "VALIDATION_ERROR",
  "data": null
}
```

### 8.2 冲突

用例：

- 重复注册同一用户名。
- 创建重复 slug 的文章。
- 创建重复 slug 的分类。
- 创建重复 name 或 slug 的标签。
- 删除仍被文章引用的分类。

期望：

```json
{
  "code": "CONFLICT",
  "data": null
}
```

### 8.3 资源不存在

用例：

- 访问不存在的文章 slug。
- 后台查看不存在的文章 ID。
- 更新不存在的评论、游客、媒体。

期望：

```json
{
  "code": "NOT_FOUND",
  "data": null
}
```

## 10. 联调冒烟脚本

以下脚本可快速验证后端是否满足前端联调基础条件。

```bash
BASE_URL=${BASE_URL:-http://localhost:8080}

echo "1. cover"
curl -s "$BASE_URL/api/cover" | jq '.code'

echo "2. admin unauthorized should be UNAUTHORIZED"
curl -s "$BASE_URL/api/admin/articles" | jq '.code'

echo "3. admin login"
ADMIN_TOKEN=$(curl -s -X POST "$BASE_URL/api/auth/login" \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"password123"}' | jq -r '.data.token')
test "$ADMIN_TOKEN" != "null" && echo "admin token ok"

echo "4. admin articles"
curl -s "$BASE_URL/api/admin/articles" \
  -H "Authorization: Bearer $ADMIN_TOKEN" | jq '.code'
```

如果以上步骤全部输出 `SUCCESS`，且未登录管理员接口输出 `UNAUTHORIZED`，说明后端基础联调可用。

## 11. 已知实现约束

- 当前后端没有 Actuator 健康检查接口。
- 当前没有专用后台分类/标签列表接口。
- 当前后端不渲染 Markdown HTML，前端需要自行渲染 `contentMarkdown`。
- 图片删除只软删除数据库记录，不物理删除文件。
- 使用 MySQL 8.4 时 Flyway 可能提示版本支持和 SQL 弃用警告，不影响当前启动和迁移。
- 如使用本地持久化数据，多次运行测试可能遇到用户名、slug、标签名重复；可换测试名或清空数据库。
